给 AI 编程助手装上「超能力」：Superpowers 安装与使用指南

发表于2026-07-09|更新于2026-07-09|AI
|总字数:2.6k|阅读时长:9分钟|浏览量:

前言

用过 Claude Code、Cursor、OpenCode 这类 AI 编程助手的同学，多半遇到过这样的场景：

你让它「加个登录功能」，它二话不说就开写，几百行代码丢出来，跑不通、漏了边界、没写测试，回头还要你一行行 review 修。

问题的根源不在于模型不够聪明，而在于没有纪律——AI 默认是「生成即祈祷」（generate-and-pray），它不会主动先想清楚再动手。

Superpowers 就是来补这一块的。它是 Jesse Vincent（obra）开源的一套面向 AI 编程助手的技能（Skills）框架，本质上是一堆精心写好的 Markdown 文档，会在合适的时机自动加载到 AI 的上下文里，把「拍脑袋写代码」改造成「先头脑风暴 → 写计划 → TDD 实现 → 代码审查」的结构化流程。

本文记录我在自己环境里装它、用它、改它的全过程。

它到底是什么

一句话：Superpowers = 给 AI 编程助手用的一套「工作流剧本」。

  • 形态：一个 GitHub 仓库，里面是一堆 SKILL.md 文件
  • 机制：通过各平台的插件系统挂到 Claude Code / OpenCode / Cursor / Codex 等 AI 助手上
  • 效果：AI 在每次响应前都会先「自查」该不该调用某个技能，比如开始写功能前先 brainstorming、写代码时坚持 test-driven-development

验证安装

不管哪个平台，装完问一句「你有哪些 superpowers？」或者「列一下你的 skills」，能列出来就 OK。Claude Code / OpenCode 还可以直接用 Skill / skill 工具手动触发来验证，比如 skill using-superpowers。

自带技能一览

装好后，仓库 skills/ 目录下自带十几个技能。挑几个核心的说说它们干啥：

技能触发时机作用
brainstorming任何创造性工作开始前苏格拉底式提问，逼你想清楚需求，产出 spec 文档。带 <HARD-GATE>——设计没定稿前一行实现代码都不让写
writing-plans设计批准后把设计拆成 2-5 分钟一颗的最小任务，每条都带文件路径和验证步骤
executing-plans计划落地阶段批量执行 + 人工 checkpoint（给不支持子 agent 的平台用）
subagent-driven-development计划落地阶段每个任务派一个全新子 agent 干活 + 两段式 review。能自己跑几小时
test-driven-development写代码时死守 RED-GREEN-REFACTOR：先写失败测试 → 看它失败 → 写最少代码 → 看它过 → commit
systematic-debugging查 bug 时4 阶段根因分析，附带 root-cause-tracing 等技术
requesting-code-review任务之间按计划比对实现，按 Critical/Important 分级报问题
using-git-worktrees开新功能前自动开 worktree 隔离 + 跑一遍干净测试基线
finishing-a-development-branch任务完成后验证测试通过，给合并/PR/丢弃几个选项
verification-before-completion声称「修好了」之前必须跑验证命令、确认输出，再下结论
writing-skills想自己写技能时技能编写指南，本身就是按 TDD 思路写的
using-superpowers每次会话启动引导技能，整个系统的「自举」入口

一条完整工作流长这样

  1. 你说「帮我加个暗色模式」
  2. brainstorming 自动触发 → 问你几个问题 → 生成 spec 文档 → <HARD-GATE> 拦住实现
  3. 你点头 → using-git-worktrees 自动触发 → 切隔离分支、跑测试基线
  4. writing-plans 自动触发 → 拆任务清单
  5. subagent-driven-development 自动触发 → 派子 agent 干活，每个任务里 test-driven-development 盯着走 TDD
  6. 任务之间 requesting-code-review 自动触发 → 查 spec 一致性 + 代码质量
  7. 全部完成后 finishing-a-development-branch 自动触发 → 给你合并/PR/丢弃选项

听起来很重，但实测对真正复杂的功能开发，能省掉大量返工。对一次性小脚本可以手动跳过。

关键机制：「响应前先查技能」

整个系统能跑起来的核心是 using-superpowers 这个引导技能。它会在每次会话启动时被注入到 AI 上下文里，内容用 <EXTREMELY_IMPORTANT> 包起来，里面有一条硬规则：

只要有 1% 的可能某个技能适用，就必须先 invoke 那个技能检查一下。

为了防止 AI 找借口跳过，还附了一张「红旗」对照表：

AI 的内心 OS现实
「这就是个简单问题」问题也是任务，先查技能
「我先探索一下代码库」技能告诉你怎么探索，先查
「这不需要正式技能」技能存在就用
「我知道这个概念」知道概念 ≠ 用了技能，invoke 它
「我先干点别的」任何动作之前先查

这套机制是 Superpowers 区别于「一堆零散 prompt」的关键——没有它，磁盘上的 SKILL.md 就是一堆死文件。

写一个自己的技能

技能就是带 YAML frontmatter 的 Markdown。位置（以 OpenCode 为例）：

  • 个人技能：~/.config/opencode/skills/<skill-name>/SKILL.md
  • 项目技能：.opencode/skills/<skill-name>/SKILL.md
  • 优先级：项目 > 个人 > Superpowers 自带

最小可用模板：

frontmatter 规则（踩过的坑）

  • 只有两个必填字段：name 和 description，合计不超过 1024 字符
  • name：只能用字母、数字、连字符，不能有括号和特殊字符
  • description：第三人称，只写「何时用」，不要写它做什么——否则 AI 会照着 description 干活而不是去读全文
  • 以「Use when...」开头最容易触发
  • 规范参考 agentskills.io/specification

ponytail：我一开始把 description 写成「这个技能会做 X、Y、Z」，结果 AI 真的照着 description 跳过了全文细节。改成「Use when...」之后才正常。

与 CLAUDE.md / AGENTS.md 的关系

很多人以为装 Superpowers 要改 CLAUDE.md，其实不用。它的硬规则是「绝不动用户的文件」（仓库 CLAUDE.md 是给贡献者看的 PR 指南，不是安装配置）。

  • CLAUDE.md：仓库贡献者指南，AI 提交 PR 要按它来（据说 AI slop 的 PR 拒绝率高达 94%）
  • AGENTS.md：软链到 CLAUDE.md，给 Codex 用
  • opencode.json：用户配置，plugins 数组里加一行就行
  • 安装器不碰你任何已有文件

关掉遥测（可选）

也认 DISABLE_TELEMETRY 和 CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC。

实战经验与坑

1. 它是按仓库作用域的。 给每个平台单独装一次，OpenCode 装了不会自动惠及 Claude Code。

2. 引导技能是一切的前提。 没它加载（<EXTREMELY_IMPORTANT> 那段），磁盘上的技能文件全是死的——这是新平台移植失败的头号原因。

3. OpenCode 的 git 插件 spec 偶发不更新。 某些 OpenCode/Bun 版本会把解析后的 git 依赖缓存住，重启不拉新提交。清掉 OpenCode 的 package 缓存或重装；或者干脆用 #v6.1.1 锁版本。

4. SDD 草稿文件从 v6.0.3 起挪到 .superpowers/sdd/。 以前在 .git/ 里（Claude Code 不让碰）。现在在工作区了——注意 git clean -fdx 会把进度记录删掉。

5. brainstorming 的 <HARD-GATE> 是真的硬。 设计没定稿前，连「简单」项目的实现代码都不让写。这是刻意的——「简单项目恰恰是没想清楚的假设浪费工时最多的地方」。想跳过就得显式跟它说「跳过设计，直接写」。

6. AI 会疯狂找借口跳过 TDD。 TDD 技能里专门有「合理化对照表」对抗这个。比如 AI 会说「事后写测试效果一样」→ 表里直接怼回去：「事后测试 = 这代码现在在干嘛？先写测试 = 这代码应该在干嘛？」

7. v6.0.0 是大重写。 subagent-driven-development 提速约 50%，token 省一半，两个 per-task reviewer 合并成一个。从 v5 升级值得。

结语

Superpowers 不是「装上就起飞」的银弹——它要求你接受「先想后写」的节奏，对那种「一句 prompt 想要一个系统」的诉求反而是阻力。但它确实把 AI 编程助手从一个「会写代码的鹦鹉」拉成了一个「有流程的初级工程师」。

如果你已经受够了 AI 写完一坨再返工，给它装一套 Superpowers，强迫它走流程，是当前最性价比的解法。
