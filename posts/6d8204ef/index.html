用 OpenSpec 给 AI 编程助手立规矩：规范驱动开发实战

发表于2026-07-09|更新于2026-07-09|AI
|总字数:2.9k|阅读时长:11分钟|浏览量:

前言

AI 编程助手最让人头疼的一点，不是它写不出代码，而是它太自信。

你说「加个用户登录」，它立刻洋洋洒洒 400 行——结果 JWT 用错了库、没考虑 token 刷新、数据库迁移没写。问题出在哪？需求只活在聊天记录里，AI 拿着半截信息就开始猜，猜错了你 400 行代码之后才发现。

OpenSpec 的口号是「最受喜爱的规范框架」（The most loved spec framework），干的事就是把**「先口头对齐要做什么」**这件事提前到写代码之前——把需求写成 markdown 规范，AI 照着规范实现，错了先改规范再改代码。

本文记录安装和走通一个完整流程的过程。

它是什么

一句话：OpenSpec = 给 AI 编程助手用的「规范驱动开发」（Spec-Driven Development, SDD）CLI + 斜杠命令系统。

  • 形态：一个 npm 全局工具 + 一组注入到各 AI 助手的斜杠命令
  • 定位：横跨 30+ AI 工具（Claude Code、Cursor、Codex、Gemini CLI、Copilot、OpenCode、Windsurf 等）
  • 哲学（README 原话）：fluid not rigid, iterative not waterfall, easy not complex, built for brownfield not just greenfield——流畅不僵化、迭代不瀑布、简单不复杂、面向存量项目不只为绿地

顺带一提：叫 "OpenSpec" 的项目不止一个。GitHub 还有个 github/spec-kit（更重）、AWS 的 kiro.dev（绑死 IDE）。本文说的是 Fission-AI/OpenSpec——面向 AI 编程助手、轻量、工具无关那个。

核心概念

上手前先把四个词记住，后面全是它们之间的流转：

概念是什么
Spec（规范）openspec/specs/ 下的 .md，描述系统当前的行为。按领域分目录（auth/、payments/、ui/），由需求（用 RFC 2119 的 SHALL/MUST/SHOULD）和场景（Given/When/Then）组成
Change（变更）一个待办改动，打包在 openspec/changes/<变更名>/，里面有 proposal.md（为什么+范围）、design.md（怎么做）、tasks.md（实现清单）、specs/（delta 规范）
Delta spec（增量规范）change 里的规范文件，用 ## ADDED Requirements / ## MODIFIED Requirements / ## REMOVED Requirements 三段描述变化。归档时：ADDED 追加、MODIFIED 替换、REMOVED 删除到主 spec
Schema工件类型和依赖的 YAML 定义，比如 spec-driven schema 规定 proposal → specs → design → tasks 的顺序，可自定义

一句话心智模型：specs/ 是真相之源（系统现在长啥样），changes/ 是待办（系统要变啥样）。归档 = 把 change 里的 delta 合并回 spec。

推荐使用顺序（工作流）

下面这套顺序是我在实战里跑下来最顺的，以「给系统加暗色模式」为例走一遍。

命令风格说明：斜杠命令的语法因平台而异——Claude Code 用冒号式 /opsx:propose，Cursor / Windsurf / Copilot / Trae 用连字符式 /opsx-propose，Kimi CLI 用 /skill:openspec-propose。本文统一用连字符式（/opsx-*），按你的平台自行替换。

0. 准备阶段（仅第一次需要）

如果你是第一次用 OpenSpec，先跑这个。它像新手教程一样带你走完一遍完整流程，对机制有个直观感受。只在第一次用，之后无需重复。

1. 构思与调研阶段

当你想做某个新功能或修 Bug、但还没具体方案时，先跑这个。它是 OpenSpec 的「无 stakes 思考搭子」——AI 读你的代码库，跟你一起权衡方案、打磨想法，不创建任何文件。文档原话：避免「AI 自信地建错东西」的最佳习惯。

2. 开启变更阶段（二选一）

想法成熟、准备动手时，挑一条路开变更：

高效路线（推荐） —— 用 /opsx-propose 或 /opsx-ff：

一步到位，直接建立变更并让 AI 把所有相关设计文档和代码制品一次性生成出来：

精细路线 —— 用 /opsx-new 开空白变更，再配合 /opsx-continue 一个接一个地生成制品：

适合你想严格控制每个工件质量、逐步推进的场景。

ponytail：默认 core profile 只给 5 个斜杠命令（explore/propose/apply/sync/archive）。上面用到的 onboard / new / continue / ff / verify / bulk-archive 都属于 expanded。跑 openspec config profile 切到 expanded，然后 openspec update 刷新 AI 技能文件。

3. 执行与实现阶段

让系统根据生成的制品去执行具体代码编写或配置调整。AI 按 tasks.md 一条条往下做，做完一条打一个勾。文档建议这一步前清一下 AI 的上下文，效果最好。

4. 验证与收尾阶段

代码或任务写完后，必须先跑这个：

它会帮你检查「实际做出来的东西」和「最初制品里设计的规格」是否一致，确保没有遗漏或偏离。手改过代码的尤其要跑——让规范和现实对账。

CLI 命令速查

终端里跑的（不是聊天里！）：

命令用途
openspec init初始化项目
openspec init --tools claude,cursor非交互指定工具
openspec update升级后刷新 AI 技能文件
openspec list列活跃 change（--specs 列规范）
openspec show <name>看详情（支持 --json）
openspec view交互式终端面板
openspec validate <name>校验结构（--all 全校）
openspec archive <name>归档
openspec status --change <name>看工件完成度
openspec instructions <artifact> --change <name>取下个工件的强化指令
openspec new change <name>CLI 直接建 change 脚手架
openspec config配置（list/get/set/unset/reset/edit/profile）
openspec schemas / schema init/fork/validate自定义 schema
openspec doctor检查 OpenSpec 根和引用健康度
openspec context拼装工作集
openspec feedback <msg>提反馈

斜杠命令在 AI 聊天里敲，语法因工具而异：

  • Claude Code：/opsx:propose（冒号）
  • Cursor / Windsurf / Copilot / Trae：/opsx-propose（连字符）
  • Kimi CLI：/skill:openspec-propose（skill 风格）

旧命令 /openspec:proposal、/openspec:apply、/openspec:archive 还能用但已废弃，建议切到 /opsx:*。

与 AI 助手的集成

openspec init 会往各工具目录里写文件——你不用手动改 CLAUDE.md：

工具技能文件命令文件
Claude Code.claude/skills/openspec-*/SKILL.md.claude/commands/opsx/<id>.md
Cursor.cursor/skills/openspec-*/SKILL.md.cursor/commands/opsx-<id>.md
Codex.codex/skills/openspec-*/SKILL.md~/.codex/prompts/opsx-<id>.md
Gemini CLI.gemini/skills/openspec-*/SKILL.md.gemini/commands/opsx/<id>.toml
GitHub Copilot.github/skills/openspec-*/SKILL.md.github/prompts/opsx-<id>.prompt.md
OpenCode.opencode/skills/openspec-*/SKILL.md.opencode/commands/opsx-<id>.md
Windsurf.windsurf/skills/openspec-*/SKILL.md.windsurf/workflows/opsx-<id>.md

生成的技能名形如 openspec-propose、openspec-explore、openspec-apply-change、openspec-archive-change。AI 在 init 时就学会了这些。升级时 openspec update 自动刷新。

配置

项目级 openspec/config.yaml（可选）：

字段作用
schema默认工作流 schema（如 spec-driven）
context注入到所有工件指令的项目上下文（≤50KB）
rules按工件类型的指导规则
store / references多仓库规划（beta）

全局配置走 openspec config list/get/set，存在 ~/.local/share/openspec/config.json。

踩过的坑与建议

1. 终端命令 ≠ 聊天命令。 这是新手第一坑。openspec ... 在终端跑；/opsx:... 在 AI 聊天里跑。不能互换。

2. 没有「进入 OpenSpec 模式」这回事。 不用启动什么交互模式，直接在 AI 聊天里敲 /opsx:propose，自动补全能弹出来就是装好了。

3. 拿不准先 /opsx:explore。 零成本思考搭子，读代码、权衡、打磨想法，不创建文件。最适合避免「AI 自信地建错东西」。

4. delta 类型别用错。 ## ADDED 新行为、## MODIFIED 改现有、## REMOVED 废弃。类型搞错 = 规范历史一坨乱。

5. 一个 change 一个意图。 一个 change 想干「顺便还要」就拆。小 change = 好审、AI session 聚焦、归档历史干净。

6. 实现前清上下文。 文档明确建议，效果差异明显。

7. 模型选高推理的。 文档推荐「Codex 5.5 和 Opus 4.7」级别。

8. openspec/ 全提交进 git。 specs、changes、archive 都是项目历史的一部分，尤其是 archive——它是「系统为什么这样」的持久证据。

9. 手改代码后跑 /opsx:verify。（expanded profile）让规范和现实对账，归档前尤其重要。

10. 工件随便改。 没有阶段门禁，实现到一半改 design.md、回头改 proposal.md 都行——AI 读的是当前文件内容。

11. 关遥测。 export OPENSPEC_TELEMETRY=0 或 export DO_NOT_TRACK=1。只收命令名 + 版本。

结语

OpenSpec 解决的不是「AI 不会写代码」，而是「AI 不知道该写什么」。它把需求对齐从聊天记录里拽出来，变成可读、可改、可归档的 markdown 规范——错的代价从「400 行代码之后」提前到了「改一段 markdown」。

配合上一篇文章的 Superpowers（管 AI 怎么干）一起用，效果更佳：OpenSpec 管做什么，Superpowers 管怎么做。
